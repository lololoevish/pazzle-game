# Adventure Puzzle Game - Project Brief

## Цели проекта

Создать одиночную 2D adventure/puzzle игру с хабом, последовательной прогрессией пещер, побочными NPC-активностями, системой сохранений и элементами визуальной новеллы.

## Текущий scope

- Основная платформа разработки: GameMaker Studio 2.
- Основной рабочий код проекта находится в `pazzle-game-gamemaker/`.
- Rust-ветка выступает как исторический референс маршрута, UX и ранних механик.
- Текущий продуктовый маршрут ориентирован на 12 пещерных уровней, хаб с тремя NPC, награды, аудио-подачу и VN-слой.

## Основные требования

### Функциональные

1. Хаб-город как центральная локация между вылазками в пещеры.
2. Основная прогрессия на 12 уровнях с разными типами головоломок.
3. Система прогресса по уровням, рычагам, золоту, предметам и побочным наградам.
4. Автосохранение и повторяемый маршрут прохождения.
5. NPC с отдельными активностями, наградами и реактивными диалогами.
6. VN-подача диалогов с печатающимся текстом и портретной системой.
7. Аудио-подсистема с музыкой по сценам и SFX по игровым событиям.

### Технические

- GameMaker Studio 2 - основная целевая среда.
- Игровой цикл и данные прогресса должны быть согласованы с 12-уровневой моделью.
- Документация и Memory Bank должны оставаться синхронизированы с реальным кодом.

## Текущее состояние проекта

- GMS2-ветка уже содержит основной каркас игры: объекты, комнаты, пазлы, аудио, сохранения и VN-скрипты.
- Проект больше не находится на стадии простого переноса структуры; текущий приоритет - runtime-стабильность, устранение legacy-расхождений и доведение маршрута до реально проходимого состояния.
- Часть документации и части GML-логики всё ещё расходятся по вопросам “6 уровней или 12”, “миграция завершена или ещё в процессе” и “какие системы считаются release-ready”.

## Ближайший рабочий фокус

1. Убрать legacy-расхождения между 6-уровневой и 12-уровневой логикой в `scr_game_state.gml` и `scr_save_system.gml`.
2. Довести основной маршрут GMS2 до состояния, где игра проходит от меню до финала без ручной правки состояния.
3. Актуализировать тестовые сценарии и ручной regression checklist под реальные объекты, комнаты и API.
4. Продолжить полировку аудио, VN-подачи и ранних тематических пещер без накопления нового архитектурного долга.

## Что уже подтверждено

- Есть 12 целевых уровней и расширенный `scr_puzzle_manager.gml`.
- Есть хаб с тремя ключевыми NPC и независимыми наградами.
- Есть отдельные GML-скрипты для аудио, сохранений, VN-подачи и прогресса.
- Есть документация и тестовые сценарии, но часть из них требует актуализации под текущий код.

## Ключевые риски

1. Несогласованность legacy-логики на 6 уровней с текущей 12-уровневой моделью.
2. Слишком оптимистичный статус части deliverables по сравнению с фактическим runtime-состоянием.
3. Большие процедурные GML-скрипты, которые затрудняют безопасное расширение без регрессий.
4. Отсутствие полноценного автоматического линтинга и формального CI-пайплайна для `.gml`.
