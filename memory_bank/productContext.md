# Adventure Puzzle Game - Product Context

## Продуктовый контекст

### Зачем нужен проект

Adventure Puzzle Game развивается как одиночная 2D adventure/puzzle игра с хабом, последовательной прогрессией по пещерам, побочными активностями NPC и выразительной сценической подачей.

Проект решает три основные задачи:

1. Дать игроку компактное, понятное и цельное приключение с головоломками, наградами и ощущением прогресса.
2. Сформировать портфолио-проект с законченным игровым маршрутом, хабом, аудио, VN-слоем и системой сохранений.
3. Поддерживать репозиторий как рабочую базу для развития GameMaker Studio 2-версии, а не как архив ранних Python/Rust-экспериментов.

### Проблемы пользователей

**Для игроков**:
- Нужна игра с понятной структурой прохождения без перегруженных систем.
- Нужен хаб, который создаёт ощущение возвращения, а не просто экран выбора уровня.
- Нужны разные типы головоломок и внятная награда за продвижение.

**Для команды разработки**:
- Нужен единый исполняемый продуктовый фокус, чтобы не распыляться между историческими ветками.
- Нужна синхронизация документации, Memory Bank и реального состояния GMS2-кода.
- Нужна управляемая 12-уровневая модель прогресса без legacy-расхождений.

### Текущее решение

- Основной продуктовый фокус закреплён за `pazzle-game-gamemaker/`.
- Rust-ветка используется как исторический референс маршрутов, механик и UX-идей.
- Целевой маршрут строится вокруг `Menu -> Town -> Cave 1..12 -> Victory -> Town`.
- Прогресс опирается на прохождение пещер, рычаги, золото, предметы, NPC-награды и сохранения.
- Диалоги развиваются в сторону VN-подачи с печатающимся текстом и портретной системой.

## Игровой опыт

### Целевая аудитория

**Игроки**:
- Любители одиночных puzzle/adventure игр.
- Пользователи, которым важны читаемый прогресс и короткие игровые сессии.
- Игроки, которым нравится сочетание хаба, небольшого сюжета и разных механик уровней.

**Команда разработки и ревьюеры**:
- Разработчики, которым нужен прозрачный и документированный GMS2-проект.
- Участники проекта, которым важно быстро понимать текущее состояние маршрута, прогресса и систем.

### Основные сценарии использования

1. Игрок запускает игру, проходит пещеры по основной линии, возвращается в хаб и продолжает маршрут через сохранение.
2. Игрок взаимодействует с NPC в хабе, получает побочные награды и дополнительные диалоги по мере прогресса.
3. Разработчик проверяет конкретную систему GMS2-версии, синхронизирует документацию и доводит маршрут до release-ready состояния.

## Ключевая ценность продукта

1. Единый продуктовый фокус на GameMaker Studio 2.
2. 12-пещерная прогрессия с разными типами головоломок.
3. Хаб с тремя NPC и побочными активностями.
4. Сохранения и повторяемый маршрут прохождения.
5. Аудио и VN-подача, усиливающие сценичность маршрута.

## Метрики успеха

- Основной маршрут от меню до финала проходим без ручной правки состояния.
- 12-уровневая модель прогресса единообразно используется в runtime-логике, UI и сохранениях.
- Документация и `memory_bank/` не расходятся с фактическим кодом GMS2-ветки.
- Побочные активности NPC не ломают основной маршрут и корректно сохраняются.

## Продуктовые ограничения

- Исторические Python/Rust-материалы больше не должны считаться основным продуктовым источником.
- Релизная готовность определяется не наличием файлов, а фактической проходимостью и стабильностью GMS2-маршрута.
- Любые изменения scope должны сначала отражаться в `docs/README.md` и `memory_bank/projectbrief.md`.

## Ближайший продуктовый фокус

1. Устранить оставшиеся legacy-расхождения между 6-уровневой и 12-уровневой логикой.
2. Довести основной GMS2-маршрут до стабильного smoke/regression-прохождения.
3. Продолжить полировку аудио, VN-диалогов и визуальной подачи хаба и пещер.
