# Adventure Puzzle Game - System Patterns

## 🎯 Архитектура

### Общая архитектура

Проект использует модульную архитектуру с двумя историческими ветками реализации, где основной исполняемой веткой сейчас является Rust:

```
Игровой цикл (main.py / main.rs)
    ↓
Сцены (scenes/)
    ├── Меню (menu.py / menu.rs)
    ├── Город (town.py / town.rs)
    ├── Геймплей (gameplay.py / gameplay.rs)
    └── Головоломки / мини-игры
    ↓
Данные прогресса и сохранения
    └── GameProgress / save_system.py
```

### Паттерны проектирования

#### 1. State Pattern (Состояние)

**Python**:
```python
class MenuScene:
    def handle_events(self, events): ...
    def update(self, keys): ...
    def draw(self, screen): ...
    def is_transition_complete(self): ...
    def get_result(self): ...
```

**Rust**:
```rust
trait Scene {
    fn handle_input(&mut self);
    fn update(&mut self);
    fn draw(&self);
    fn get_next_state(&self) -> Option<GameState>;
    fn take_completed_level(&mut self) -> Option<u8> { None }
    fn take_progress_update(&mut self) -> Option<ProgressUpdate> { None }
}
```

**Применение**: управление состояниями игры и переключение между `Menu`, `Town`, `Playing`, `Quit`.

#### 2. Модульная изоляция головоломок

Каждая Rust-головоломка живёт в отдельном модуле и подключается в `GameplayScene` по номеру уровня.

**Применение**: локализация логики уровня, упрощение поддержки и расширения.

#### 3. Одноразовый сигнал завершения уровня

Трейт `Scene` содержит `take_completed_level()`, чтобы сцена один раз сообщила главному циклу о завершении уровня.

**Применение**: синхронизация прогресса без постоянного опроса внутреннего состояния пазла.

#### 4. Точечные обновления прогресса

`GameplayScene` использует `take_progress_update()` для сохранения состояния рычага без прямого доступа сцены к файловой системе.

**Применение**: хранение `lever_pulled` в `savegame.json`, когда игрок открыл проход в следующую пещеру.

#### 5. Централизованный менеджер переходов

`main.rs` содержит отдельное состояние перехода (`FadeOut` / `FadeIn`) и накладывает общий анимированный оверлей поверх любой сцены.

**Применение**: сцены остаются сфокусированы на своём вводе, логике и отрисовке, а смена `GameState` получает единый визуальный язык без копирования кода между сценами.

#### 6. Диалоговый оверлей NPC

`TownScene` хранит активный диалог отдельно от статусной строки и временно переключает обработку ввода в режим разговора.

**Применение**: NPC могут иметь несколько последовательных реплик, печатающийся текст и собственный UI-слой, не смешанный с обычными подсказками хаба.

#### 7. Реактивные реплики по прогрессу

`TownScene` собирает текст реплик из текущего `GameProgress`: активной пещеры, числа открытых печатей и отдельных флагов завершения.

**Применение**: NPC не повторяют один и тот же текст на всём протяжении игры, а комментируют актуальную фазу прохождения.

#### 8. Разделение world UI и puzzle UI

`GameplayScene` при входе в головоломку перестаёт рисовать верхнюю и нижнюю панели мира и вместо них показывает отдельную рамку пазла с собственным заголовком, инструкциями и статусом.

**Применение**: тексты головоломок не накладываются на HUD пещеры, а разные режимы сцены читаются как разные визуальные состояния.

#### 9. Локальный модуль визуальных ассетов

`visual_assets.rs` лениво создаёт `Texture2D` из локальных PNG через `include_bytes!` и отдаёт сцены готовые текстуры по роли и направлению.

**Применение**: пиксельные спрайты игрока, NPC, рычага и артефакта можно использовать без асинхронной загрузки и без ручного копирования кода инициализации по сценам.

#### 10. Разделение постоянного прогресса и локального состояния двери

`lever_pulled` в `GameProgress` хранит факт открытия следующего уровня, но не заставляет дверь в повторно открытой сцене оставаться открытой навсегда.

**Применение**: при повторном входе в уже решённую пещеру игрок видит закрытую дверь, может либо перепройти пазл, либо сразу дёрнуть рычаг и открыть проход заново.

#### 11. Подтверждённый сброс прохождения

`MenuScene` открывает отдельный confirm-оверлей для новой игры, а `main.rs` обрабатывает специальное состояние `GameState::ResetGame`.

**Применение**: сброс прохождения удаляет/пересоздаёт `savegame.json`, обнуляет `GameProgress` и запускает игру заново без ручного вмешательства в файлы.

#### 12. Прогресс как часть интерфейса

`MenuScene` и `TownScene` читают `GameProgress` не только для маршрутов, но и для прямой визуализации состояния прохождения.

**Применение**: игрок видит число открытых печатей, число решённых уровней и текущую цель экспедиции ещё до входа в пещеру, что уменьшает потерю контекста после возврата в меню или хаб.

#### 13. Динамический action-label в меню

`MenuScene` меняет подпись первого действия в зависимости от сохранения: новый профиль получает «Играть», а существующий прогресс — «Продолжить».

**Применение**: меню сразу сообщает, будет ли действие начинать новую экспедицию или продолжать уже существующую.

## 🔄 Потоки данных

### Игровой цикл

`main.rs` загружает сохранение, создаёт активную сцену и в цикле вызывает:

1. `handle_input()`
2. `update()`
3. `take_completed_level()`
4. `take_progress_update()`
5. сохранение прогресса при необходимости
6. `draw()`
7. запуск централизованного fade-перехода при наличии `get_next_state()`
8. замена сцены и fade-in через менеджер переходов

### Обработка ввода

**Python**:
```python
for event in pygame.event.get():
    if event.type == pygame.QUIT:
        running = False
    elif event.type == pygame.KEYDOWN:
        # Обработка нажатия клавиш
```

**Rust**:
```rust
if is_key_pressed(KeyCode::Escape) {
    // Обработка нажатия ESC
}
```

### Отрисовка

**Python**:
```python
screen.fill(BLACK)
current_scene.draw(screen)
pygame.display.flip()
```

**Rust**:
```rust

next_frame().await
```

## 🎮 Игровая логика

### Состояния игры

```
GameState
├── Menu
├── Town (стартовый хаб и шахтный спуск)
├── Playing(level)
│   ├── 1 -> Maze
│   ├── 2 -> WordSearch
│   ├── 3 -> Pattern
│   ├── 4 -> MemoryMatch
│   ├── 5 -> Platformer
│   └── 6 -> FinalChallenge
└── Quit
```

### Система сохранений

**Формат**: JSON
**Расположение**: в рабочей директории приложения
**Содержимое**:
```json
{
  "levels": {
    "1": {"completed": true, "lever_pulled": false},
    "2": {"completed": false, "lever_pulled": false}
  },
  "gold": 100,
  "items": []
}
```

### Система прогрессии

1. Игрок из `TownScene` входит в шахтный спуск и попадает в текущую активную пещеру.
1. Игрок из `TownScene` входит в шахтный спуск и всегда стартует с первой пещеры цепочки.
2. Внутри `GameplayScene` пещера содержит алтарь головоломки, рычаг и физический проход в следующую пещеру.
3. После первого решения печати сцена одноразово сообщает о завершении уровня через `take_completed_level()`.
4. После опускания рычага сцена сообщает о `lever_pulled` через `take_progress_update()`, и только это открывает следующий уровень.
5. Повторный вход в уже решённую головоломку позволяет нажать `L` и пропустить повторное решение перед рычагом.
6. `TownScene` использует сохранённый прогресс для подсказок, диалогов и визуализации статуса шести печатей, но не для прямого прыжка в глубину цепочки.
7. Для уровня 1 движение реализовано как скольжение до ближайшей стены.
8. Для уровня 2 `GameplayScene` использует последовательные фазы: сначала `WordSearchPuzzle`, затем `MemoryMatchPuzzle`.

## 🛠️ Технические решения

### Rust

- **Язык**: Rust 1.70+
- **Движок**: macroquad 0.4
- **Сериализация**: serde + serde_json
- **Рандом**: rand 0.8
- **Формат сохранений**: JSON
- **Ключевая модель**: `Scene` trait + `GameState` enum + отдельные puzzle modules

## 🎯 Паттерны использования

### Добавление нового уровня в Rust

1. Создать модуль в `src_rust/scenes/puzzles/`
2. Реализовать `new()`, `handle_input()`, `update()`, `draw()`, `is_solved()`
3. Подключить модуль в `src_rust/scenes/puzzles/mod.rs`
4. Добавить выбор пазла в `src_rust/scenes/gameplay.rs`
5. При необходимости обновить тексты и маршрут шахтного спуска в `src_rust/scenes/town.rs`

### Добавление NPC-логики в Rust

1. Расширить `TownScene` визуальным и интерактивным объектом
2. При необходимости подключить отдельный диалоговый или мини-игровой оверлей
3. Зафиксировать новые маршруты и UI-паттерны в `docs/README.md` и `memory_bank/`
