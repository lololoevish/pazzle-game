# Adventure Puzzle Game - System Patterns

## Архитектура

### Общая архитектура

Основной GMS2 runtime проекта находится в `pazzle-game-gamemaker/`. Rust-ветка сохраняется как исторический референс маршрута и прошлых UX-решений, но не определяет текущую архитектурную правду. Для запуска без GameMaker добавлен отдельный Phaser 3 web-port в `web-port/`.

### Основной игровой маршрут

Текущий целевой маршрут GMS2-версии:

`Menu -> Town -> Cave 1..24 -> Victory -> Town`

Ключевые инварианты:

1. Хаб остаётся центральной точкой возврата между пещерами.
2. Основная прогрессия строится вокруг 24 пещер и рычагов.
3. Побочные NPC-активности не должны блокировать или ломать основной маршрут.
4. Повторный вход в завершённую пещеру должен быть предсказуемым для прогресса и сохранений.

## Паттерны

### 1. Game State как центральная модель прогресса

`scr_game_state.gml` хранит и обновляет состояние 24 уровней, текущую цель экспедиции, completed/lever flags и общий статус маршрута.

Применение: все системы, которые читают прогресс, должны исходить из 24-уровневой модели и избегать локальных хардкодов количества уровней.

### 2. Save System как слой восстановления состояния

`scr_save_system.gml` отвечает за сохранение, загрузку, reset и расширение legacy-сохранений до текущей длины `SAVE_LEVEL_COUNT = 24`.

Применение: сохранение должно безопасно восстанавливать уровни, золото, предметы и NPC-награды без ручной правки JSON/INI.

### 3. Разделение world UI и puzzle UI

`scr_ui_manager.gml` отвечает за общие UI-оверлеи, HUD, уведомления, диалоги и сервисные сообщения. Puzzle scripts рисуют собственные панели и инструкции только внутри активной механики.

Применение: UI мира, пазлов и диалогов не должен конфликтовать по состояниям ввода и отрисовки.

### 4. Изоляция puzzle scripts

Каждая головоломка живёт в отдельном GML-скрипте внутри `scripts/puzzle_scripts/`, а `scr_puzzle_manager.gml` маршрутизирует тип пазла по уровню.

Применение: добавление или исправление одной головоломки не должно требовать переписывания общего game state или save system.

### 5. Хабовые NPC как независимые активности

`obj_npc.gml` содержит побочные активности, реактивные состояния NPC и интеграцию с VN-слоем.

Применение: награды NPC сохраняются в общей модели прогресса, но не являются обязательным условием прохождения 24 пещер.

### 6. VN-слой поверх NPC-диалогов

`scr_vn_portrait_manager.gml`, `scr_vn_text_engine.gml`, `scr_vn_dialogue_system.gml`, `scr_vn_ui_manager.gml` и `scr_vn_master.gml` формируют слой визуальной новеллы.

Применение: VN-система должна gracefully degrade при отсутствии портретов или необязательных ресурсов.

### 7. Аудио как fail-safe подсистема

`scr_audio_manager.gml` выбирает музыку по комнатам/состояниям, проигрывает SFX и хранит контекстные модификаторы громкости.

Применение: отсутствие звукового ресурса не должно приводить к runtime-падению; функция возвращает `-1` и пропускает воспроизведение.

### 8. Переходы между уровнями без конфликта имён

`scr_level_transition_platformer.gml` обслуживает последовательные переходы между пещерами. `scr_interlevel_platformer.gml` обслуживает межуровневые точки перехода из комнат.

Применение: функции перехода имеют разные имена и разные структуры входных данных, чтобы не перетирать друг друга в глобальной области GML.

### 9. Runtime-safe optional globals

Необязательные глобальные ресурсы проверяются через `variable_global_exists()` перед чтением.

Применение: аудио, портреты, UI-шрифты, mercy-прогресс и временный ввод не должны падать, если соответствующий global ещё не создан.

### 10. Memory Bank как операционный контекст

`memory_bank/` фиксирует текущую архитектурную и продуктовую правду для агента. `docs/README.md` остаётся единственным источником верхнеуровневой архитектуры, а `memory_bank/projectbrief.md` остаётся каноническим источником deliverables и процента выполнения.

### 11. Web-port как self-contained runnable copy

`web-port/` изолирован от GMS2-кода и реализует собственные Phaser scenes, TypeScript game state и localStorage save system. Пользовательские PNG подключаются через `public/sprites/manifest.json`; отсутствие ассета не ломает runtime, потому что `SpriteManifest.ts` создаёт placeholder texture. Пещерные пазлы строятся вокруг героя: игрок двигает персонажа к объектам и нажимает `E`/`Space`, вместо прямого клика по UI-элементам.

Хаб и пещеры web-port используют Arcade Physics static walls для запрета прохождения сквозь препятствия. Переходы между локациями реализованы как overlap-зоны: герой заходит в проход, и сцена меняется автоматически без дополнительного нажатия.

Применение: web-port нужен для быстрого запуска и итерации без GameMaker Studio 2, но не должен неявно менять GMS2 runtime-контракты.

## Текущие технические риски

1. GML-скрипты крупных систем остаются монолитными.
2. Автоматический линтинг `.gml` отсутствует, потому что текущая конфигурация Biome игнорирует эти файлы.
3. Runtime-проверка требует ручного запуска в GameMaker Studio 2.
4. Часть исторических документов описывает прошлые Python/Rust этапы и должна читаться как legacy reference, если не синхронизирована с `docs/README.md`.
5. Web-port пока содержит прототипы пазлов и требует расширения ассетного manifest, аудио и VN-подачи для соответствия полному GMS2 scope.

## Следующие архитектурные шаги

1. Усилить `scr_save_system.gml` против повреждённых JSON/INI и отсутствующих полей.
2. Декомпозировать крупные GML-скрипты только после стабилизации smoke-прохода.
3. Проверить runtime-проход `Menu -> Town -> Cave 1..24 -> Victory -> Town` в GameMaker Studio 2.
4. Синхронизировать оставшиеся исторические документы или явно пометить их как legacy reference.
5. Расширить web-port: подключить реальные спрайты, фоны/тайлы, аудио и заменить прототипы пазлов полноценными версиями.
