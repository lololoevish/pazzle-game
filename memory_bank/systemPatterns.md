# Adventure Puzzle Game - System Patterns

## 🎯 Архитектура

### Общая архитектура

Проект использует модульную архитектуру с двумя историческими ветками реализации, где основной исполняемой веткой сейчас является Rust:

```
Игровой цикл (main.py / main.rs)
    ↓
Сцены (scenes/)
    ├── Меню (menu.py / menu.rs)
    ├── Город (town.py / town.rs)
    ├── Геймплей (gameplay.py / gameplay.rs)
    └── Головоломки / мини-игры
    ↓
Данные прогресса и сохранения
    └── GameProgress / save_system.py
```

### Паттерны проектирования

#### 1. State Pattern (Состояние)

**Python**:
```python
class MenuScene:
    def handle_events(self, events): ...
    def update(self, keys): ...
    def draw(self, screen): ...
    def is_transition_complete(self): ...
    def get_result(self): ...
```

**Rust**:
```rust
trait Scene {
    fn handle_input(&mut self);
    fn update(&mut self);
    fn draw(&self);
    fn get_next_state(&self) -> Option<GameState>;
    fn take_completed_level(&mut self) -> Option<u8> { None }
    fn take_progress_update(&mut self) -> Option<ProgressUpdate> { None }
}
```

**Применение**: управление состояниями игры и переключение между `Menu`, `Town`, `Playing`, `Victory`, `Quit`.

#### 2. Модульная изоляция головоломок

Каждая Rust-головоломка живёт в отдельном модуле и подключается в `GameplayScene` по номеру уровня.

**Применение**: локализация логики уровня, упрощение поддержки и расширения.

#### 3. Одноразовый сигнал завершения уровня

Трейт `Scene` содержит `take_completed_level()`, чтобы сцена один раз сообщила главному циклу о завершении уровня.

**Применение**: синхронизация прогресса без постоянного опроса внутреннего состояния пазла.

#### 4. Точечные обновления прогресса

`GameplayScene` использует `take_progress_update()` для сохранения состояния рычага без прямого доступа сцены к файловой системе.

**Применение**: хранение `lever_pulled` в `savegame.json`, когда игрок открыл проход в следующую пещеру.

#### 5. Централизованный менеджер переходов

`main.rs` содержит отдельное состояние перехода (`FadeOut` / `FadeIn`) и накладывает общий анимированный оверлей поверх любой сцены.

**Применение**: сцены остаются сфокусированы на своём вводе, логике и отрисовке, а смена `GameState` получает единый визуальный язык без копирования кода между сценами.

#### 6. Диалоговый оверлей NPC

`TownScene` хранит активный диалог отдельно от статусной строки и временно переключает обработку ввода в режим разговора.

**Применение**: NPC могут иметь несколько последовательных реплик, печатающийся текст и собственный UI-слой, не смешанный с обычными подсказками хаба.

#### 7. Реактивные реплики по прогрессу

`TownScene` собирает текст реплик из текущего `GameProgress`: активной пещеры, числа открытых печатей и отдельных флагов завершения.

**Применение**: NPC не повторяют один и тот же текст на всём протяжении игры, а комментируют актуальную фазу прохождения.

#### 8. Разделение world UI и puzzle UI

`GameplayScene` при входе в головоломку перестаёт рисовать верхнюю и нижнюю панели мира и вместо них показывает отдельную рамку пазла с собственным заголовком, инструкциями и статусом.

**Применение**: тексты головоломок не накладываются на HUD пещеры, а разные режимы сцены читаются как разные визуальные состояния.

#### 9. Локальный модуль визуальных ассетов

`visual_assets.rs` лениво создаёт `Texture2D` из локальных PNG через `include_bytes!` и отдаёт сцены готовые текстуры по роли и направлению.

**Применение**: пиксельные спрайты игрока, NPC, рычага и артефакта можно использовать без асинхронной загрузки и без ручного копирования кода инициализации по сценам.

#### 10. Разделение постоянного прогресса и локального состояния двери

`lever_pulled` в `GameProgress` хранит факт открытия следующего уровня, но не заставляет дверь в повторно открытой сцене оставаться открытой навсегда.

**Применение**: при повторном входе в уже решённую пещеру игрок видит закрытую дверь, может либо перепройти пазл, либо сразу дёрнуть рычаг и открыть проход заново.

#### 11. Подтверждённый сброс прохождения

`MenuScene` открывает отдельный confirm-оверлей для новой игры, а `main.rs` обрабатывает специальное состояние `GameState::ResetGame`.

**Применение**: сброс прохождения удаляет/пересоздаёт `savegame.json`, обнуляет `GameProgress` и запускает игру заново без ручного вмешательства в файлы.

#### 12. Прогресс как часть интерфейса

`MenuScene` и `TownScene` читают `GameProgress` не только для маршрутов, но и для прямой визуализации состояния прохождения.

**Применение**: игрок видит число открытых печатей, число решённых уровней и текущую цель экспедиции ещё до входа в пещеру, что уменьшает потерю контекста после возврата в меню или хаб.

#### 13. Динамический action-label в меню

`MenuScene` меняет подпись первого действия в зависимости от сохранения: новый профиль получает «Играть», а существующий прогресс — «Продолжить».

**Применение**: меню сразу сообщает, будет ли действие начинать новую экспедицию или продолжать уже существующую.

#### 14. Отдельная сцена завершения экспедиции

После шестой пещеры переход идёт не в хаб напрямую, а в `VictoryScene`, которая показывает итог прохождения и только затем возвращает игрока в деревню или меню.

**Применение**: полный цикл прогрессии получает явный финал, а завершение игры отделяется от обычного возврата в хаб.

#### 15. Побочная NPC-мини-игра с постоянной наградой

`TownScene` теперь умеет работать не только как диалоговый хаб, но и как контейнер отдельной мини-игры механика Роана. Сцена временно переключает ввод в режим калибровки, а завершение отправляет `ProgressUpdate` в основной цикл.

**Применение**: побочные активности NPC остаются в общей архитектуре сцен, но при этом награды сохраняются централизованно в `GameProgress`, а не внутри UI-сцены.

#### 16. Несколько независимых хаб-активностей в одной сцене

`TownScene` одновременно содержит несколько взаимоисключающих overlay-режимов: диалог, калибровку механика и викторину архивариуса. Каждый режим забирает ввод только на время своей активности и не смешивается с остальным UI.

**Применение**: хаб расширяется новыми активностями без выделения отдельных сцен, но при этом режимы остаются изолированными и не конфликтуют по управлению.

#### 17. Испытание старосты как числовая игра на выдержку

Староста Иара использует отдельный overlay-режим угадывания числа с ограниченным числом попыток. Состояние игры живёт внутри `TownScene`, а награда сохраняется через отдельный `ProgressUpdate`.

**Применение**: хаб получает третий тип побочной активности, который отличается и от реакции, и от викторины, но остаётся встроенным в тот же маршрут и UI-каркас.

#### 18. Chapter-подача меню и хаба

`MenuScene` и `TownScene` используют не просто служебные панели, а устойчивый визуальный язык: крупные титулы, chapter-лейблы, плотные контрастные карточки, акцентные ореолы света и отдельную фокус-панель для ближайшей интерактивной точки.

**Применение**: меню и стартовый хаб читаются как сцены с драматургией кадра, а не как нейтральные debug-экраны. Это упрощает дальнейшее выравнивание поздних пещер и финала под тот же стиль.

#### 19. Наградо-зависимые реплики NPC

`TownScene` разделяет не только общий прогресс по печатям, но и локальное состояние каждой побочной активности: до награды, после награды и после финала экспедиции. Preview в фокус-панели и полный диалог используют один и тот же контекст награды.

**Применение**: NPC ощущаются не как статические справочники, а как жители, которые помнят уже выданные игроку предметы и меняют тон разговора после завершения своей линии.

#### 20. Единый chapter-style язык для пещер и финала

`GameplayScene` использует level-specific акцентные цвета, chapter-лейблы и усиленный puzzle-shell, а `VictoryScene` оформляет summary-экран в том же визуальном языке. Стиль больше не ограничен меню и хабом.

**Применение**: маршрут `Menu -> Town -> Playing -> Victory` воспринимается как единый последовательный спектакль, а не как набор визуально разрозненных экранов.

#### 21. Локальная SFX-подсистема без внешнего пайплайна

`audio.rs` инициализирует небольшой набор встроенных WAV через `include_bytes!` и `macroquad::audio`. Сцены вызывают короткие helper-функции для навигации, подтверждения, отмены, успеха и рычага.

**Применение**: игра получила базовую звуковую обратную связь без зависимости от внешнего загрузчика ассетов, отдельного аудиоменеджера или ручного размещения путей по сценам.

#### 22. Общая оболочка пещеры + level-specific декоративный слой

`GameplayScene` держит общую геометрию пещеры, алтаря, двери и HUD в одном наборе методов, а поверх него добавляет отдельный декоративный слой по номеру уровня.

**Применение**: поздние пещеры можно делать визуально разными через локальные мотивы окружения, не дробя сцену на шесть отдельных реализаций с дублированием одной и той же логики.

#### 23. SFX привязаны к игровым событиям пазлов, а не только к смене сцен

Пазлы используют тот же `audio.rs`, что и хаб со сценами, но вызывают звуки в собственных точках состояния: правильный ввод, ошибка, подбор коллектаблов, падение, столкновение и тайм-аут.

**Применение**: звуковая обратная связь воспринимается как часть самой механики уровня, а не только как оболочка меню и переходов.

#### 24. Музыка переключается централизованно по состоянию сцены

`audio.rs` теперь хранит не только SFX, но и loop-треки по сценам, а `main.rs` вызывает единый `sync_music(state)` при создании новой сцены.

**Применение**: меню, деревня, пещеры и финал получают разные музыкальные состояния без копирования логики по сценам и без постоянного перезапуска одного и того же трека каждый кадр.

#### 25. Тематические декор-спрайты поздних пещер подключаются через общий ассет-модуль

`visual_assets.rs` расширен отдельными текстурами окружения для уровней 3-6, а `GameplayScene` использует их поверх общего пещерного слоя и поверх процедурного света/линий.

**Применение**: проект остаётся в одной сцене и одном рендер-пайплайне, но поздние уровни получают более ощутимую материальность за счёт реальных PNG-ассетов, а не только примитивов.

#### 26. Меню зафиксировано под реальную высоту окна

`MenuScene` держит summary-панель, статус сохранения, четыре action-card и нижний hint в одной вертикальной сетке, рассчитанной под 800x600.

**Применение**: состояние завершённой экспедиции с длинным summary не ломает layout и не уводит нижние кнопки под футер.

#### 27. Аудио работает по fail-safe модели

`audio.rs` оборачивает вызовы `play_sound` и `stop_sound` в защиту от panic. Если аудиобэкенд системы не поддерживает runtime-воспроизведение, модуль отключает звук до конца сессии вместо падения процесса.

**Применение**: игра остаётся запускаемой на проблемных машинах, даже если музыка и SFX временно недоступны.

#### 28. План миграции на GameMaker 2

Проект включает детальный план миграции на GameMaker 2, описанный в `plans/gamemaker_migration_plan.md`. Архитектура GameMaker включает:

- `obj_game_manager` - управление состоянием игры
- `obj_player` - игрок и его поведение  
- `obj_puzzle` - базовый класс для головоломок
- `obj_lever` - объект рычага для прогресса
- `obj_npc` - персонажи в хабе
- `obj_interactable` - интерактивные объекты
- Скрипты: `scr_game_state`, `scr_save_system`, `scr_audio_manager`, `scr_ui_manager`
- Puzzle-скрипты: `scr_maze_puzzle`, `scr_word_search_puzzle`, `scr_rhythm_puzzle`, `scr_memory_match_puzzle`, `scr_platformer_puzzle`, `scr_final_challenge_puzzle`
- Менеджер: `scr_puzzle_manager`
- Комнаты: `rm_menu`, `rm_town`, `rm_cave_maze`, `rm_cave_archive`, `rm_cave_rhythm`, `rm_cave_pairs`, `rm_cave_platformer`, `rm_cave_final`, `rm_victory`

**Применение**: позволяет перенести всю функциональность из Rust-версии в GameMaker с сохранением игровой механики и контента.

#### 29. Структура ассетов GameMaker

Ассеты проекта GameMaker организованы в следующие категории:

- `assets/sprites/` - спрайты для персонажей, объектов и интерфейса
- `assets/sounds/` - звуковые эффекты для UI и игровых событий
- `assets/music/` - музыкальные темы для разных сцен
- `assets/fonts/` - шрифты для интерфейса и текстов
- `assets/effects/` - визуальные эффекты

**Применение**: обеспечивает структурированное хранение ресурсов и упрощает импорт/замены ассетов.

#### 30. Документация тестирования GameMaker проекта

Для обеспечения качества функциональности проекта созданы документы:

- `TESTING.md` - сценарии интеграционного тестирования для проверки работы всех игровых компонентов
- `UNIT_TESTS.md` - сценарии модульного тестирования для проверки отдельных функций и систем

**Применение**: позволяет проверить корректность реализации всех игровых механик и систем, обеспечивая соответствие функциональности оригинальной Rust-версии.

#### 31. Сценарии модульного тестирования GameMaker проекта

Для проверки корректности функционирования отдельных компонентов реализованы сценарии модульного тестирования:

- `test_scenarios/test_game_state.gml` - тестирование системы состояния игры
- `test_scenarios/test_save_system.gml` - тестирование системы сохранений

- `test_scenarios/test_ui_manager.gml` - тестирование UI-менеджера

**Применение**: позволяет проверить корректность работы отдельных модулей без зависимости от других компонентов системы.

#### 32. Миграция с заглушек на рабочие компоненты

Для обеспечения полноценной работоспособности проекта выполнена замена текстовых заглушек на функциональные GML-скрипты:

- Обновлен `scr_audio_manager.gml` - реализована полноценная система аудио с обработкой отсутствующих ресурсов
- Обновлен `scr_game_state.gml` - добавлены расширенные функции управления прогрессом игры
- Обновлен `scr_save_system.gml` - реализована корректная система сохранения/загрузки с обработкой ошибок
- Обновлен `scr_ui_manager.gml` - добавлена надежная система UI с инициализацией и обработкой данных

**Применение**: обеспечивает работоспособность всех основных компонентов системы без необходимости в реальных аудио и графических ресурсах на этапе разработки.

#### 33. Структура файлов спрайтов GameMaker

Для корректной работы визуальных компонентов созданы файлы конфигурации спрайтов:

- `spr_player.sprite2` - спрайт игрока с анимациями движения
- `spr_npc.sprite2` - спрайты NPC с анимациями для трех персонажей
- `spr_lever.sprite2` - спрайт рычага с состояниями (нормальный, опущенный)
- `spr_key_objects.sprite2` - спрайты ключевых игровых объектов (кристаллы, артефакты, двери)
- `spr_background.sprite2` - фоны для различных сцен (меню, город, пещеры, финал)
- `spr_ui_sprites.sprite2` - UI элементы (кнопки, подсказки, индикаторы прогресса)

**Применение**: обеспечивает структурированное хранение информации о спрайтах, позволяя правильно отображать визуальные компоненты игры.

`main.rs` загружает сохранение, создаёт активную сцену и в цикле вызывает:

1. `handle_input()`
2. `update()`
3. `take_completed_level()`
4. `take_progress_update()`
5. сохранение прогресса при необходимости

7. запуск централизованного fade-перехода при наличии `get_next_state()`

## 🎮 Игровая логика

### Состояния игры

```
GameState
├── Menu
├── Town (стартовый хаб и шахтный спуск)
├── Playing(level)
│   ├── 1 -> Maze
│   ├── 2 -> WordSearch
│   ├── 3 -> Pattern
│   ├── 4 -> MemoryMatch
│   ├── 5 -> Platformer
│   └── 6 -> FinalChallenge
├── Victory
└── Quit
```

### Система сохранений

**Формат**: JSON
**Расположение**: в рабочей директории приложения
**Содержимое**:
```json
{
  "levels": {
    "1": {"completed": true, "lever_pulled": false},
    "2": {"completed": false, "lever_pulled": false}
  },
  "gold": 100,
  "items": [],
  "elder_trial_completed": false,
  "mechanic_training_completed": false,
  "archivist_quiz_completed": false
}
```

### Система прогрессии

1. Игрок из `TownScene` входит в шахтный спуск и попадает в текущую активную пещеру.
1. Игрок из `TownScene` входит в шахтный спуск и всегда стартует с первой пещеры цепочки.
2. Внутри `GameplayScene` пещера содержит алтарь головоломки, рычаг и физический проход в следующую пещеру.
3. После первого решения печати сцена одноразово сообщает о завершении уровня через `take_completed_level()`.
4. После опускания рычага сцена сообщает о `lever_pulled` через `take_progress_update()`, и только это открывает следующий уровень.
5. Повторный вход в уже решённую головоломку позволяет нажать `L` и пропустить повторное решение перед рычагом.
6. `TownScene` использует сохранённый прогресс для подсказок, диалогов и визуализации статуса шести печатей, но не для прямого прыжка в глубину цепочки.
7. Для уровня 1 движение реализовано как скольжение до ближайшей стены.
8. Для уровня 2 `GameplayScene` использует последовательные фазы: сначала `WordSearchPuzzle`, затем `MemoryMatchPuzzle`.
9. После опускания рычага шестой пещеры сцена переводит маршрут в `VictoryScene`, а состояние полного прохождения определяется по финальному `lever_pulled`.
10. В деревне староста Иара предлагает отдельное испытание на угадывание числа; первое успешное прохождение сохраняет награду через `ProgressUpdate::ElderTrialCompleted`.
11. В деревне механик Роан предлагает отдельную калибровку на стрелках; первое успешное прохождение сохраняет награду через `ProgressUpdate::MechanicTrainingCompleted`.
12. В деревне архивариус Тель предлагает викторину на знание правил экспедиции; первое успешное прохождение сохраняет награду через `ProgressUpdate::ArchivistQuizCompleted`.

## 🎯 Паттерны использования

### Добавление нового уровня в Rust

1. Создать модуль в `src_rust/scenes/puzzles/`
2. Реализовать `new()`, `handle_input()`, `update()`, `draw()`, `is_solved()`
3. Подключить модуль в `src_rust/scenes/puzzles/mod.rs`
4. Добавить выбор пазла в `src_rust/scenes/gameplay.rs`
5. При необходимости обновить тексты и маршрут шахтного спуска в `src_rust/scenes/town.rs`

### Добавление NPC-логики в Rust

1. Расширить `TownScene` визуальным и интерактивным объектом
2. При необходимости подключить отдельный диалоговый или мини-игровой оверлей
3. Зафиксировать новые маршруты и UI-паттерны в `docs/README.md` и `memory_bank/`
4. Если меняется визуальный язык хаба или меню, синхронизировать это как архитектурный UX-паттерн, а не только как changelog-запись.
