# Документ сценариев модульного тестирования GameMaker 2 проекта

## Общая информация

Этот документ описывает сценарии модульного тестирования для проверки корректности функционирования отдельных компонентов GameMaker 2 версии "Adventure Puzzle Game".

## Сценарии модульного тестирования

### 1. Система состояния игры (scr_game_state.gml)

#### Тест 1.1: Инициализация состояния
- **Функция**: `scr_game_state_init()`
- **Входные данные**: none
- **Ожидаемый результат**: Установка начального состояния игры как MENU
- **Проверки**:
  - `global.current_state == STATE_MENU`
  - `global.game_initialized == true`

#### Тест 1.2: Переключение состояний
- **Функция**: `scr_set_game_state(new_state)`
- **Входные данные**: STATE_HUB, STATE_LEVEL, STATE_PAUSE, STATE_DIALOG
- **Ожидаемый результат**: Корректное изменение состояния игры
- **Проверки**:
  - `global.current_state == new_state`
  - Вызов соответствующих функций инициализации нового состояния

#### Тест 1.3: Получение текущего состояния
- **Функция**: `scr_get_game_state()`
- **Входные данные**: none
- **Ожидаемый результат**: Возврат текущего состояния игры
- **Проверки**:
  - Возвращаемое значение соответствует `global.current_state`

### 2. Система сохранений (scr_save_system.gml)

#### Тест 2.1: Инициализация системы сохранений
- **Функция**: `scr_save_system_init()`
- **Входные данные**: none
- **Ожидаемый результат**: Подготовка структуры данных для сохранений
- **Проверки**:
  - Создание структуры `global.save_data`
  - Установка начальных значений прогресса

#### Тест 2.2: Сохранение данных
- **Функция**: `scr_save_game(slot)`
- **Входные данные**: номер слота (0-9)
- **Ожидаемый результат**: Сохранение данных в указанный слот
- **Проверки**:
  - Данные сохранены в соответствующем файле
  - Структура сохранения корректна
  - Обработка ошибок при невозможности сохранения

#### Тест 2.3: Загрузка данных
- **Функция**: `scr_load_game(slot)`
- **Входные данные**: номер слота (0-9)
- **Ожидаемый результат**: Загрузка данных из указанного слота
- **Проверки**:
  - Данные корректно загружены
  - Структура данных восстановлена
  - Обработка случая отсутствия сохранения

#### Тест 2.4: Проверка наличия сохранения
- **Функция**: `scr_save_exists(slot)`
- **Входные данные**: номер слота (0-9)
- **Ожидаемый результат**: Возврат true/false в зависимости от наличия сохранения
- **Проверки**:
  - Возвращаемое значение корректно

### 3. Аудио-менеджер (scr_audio_manager.gml)

#### Тест 3.1: Инициализация аудио-менеджера

#### Тест 3.2: Воспроизведение музыки
- **Функция**: `scr_play_music(music_asset, loop)`
- **Входные данные**: ресурс музыки, флаг зацикливания
- **Ожидаемый результат**: Запуск воспроизведения музыки
- **Проверки**:
  - Музыка воспроизводится
  - Учет флага зацикливания
  - Обработка отсутствия аудиоресурса

#### Тест 3.3: Воспроизведение звукового эффекта

- **Входные данные**: ресурс SFX
- **Ожидаемый результат**: Воспроизведение звукового эффекта
- **Проверки**:
  - Звук воспроизводится
  - Обработка отсутствия аудиоресурса

#### Тест 3.4: Управление громкостью
- **Функция**: `scr_set_volume(channel, volume)`
- **Входные данные**: канал (music/sfx), значение громкости (0-1)
- **Ожидаемый результат**: Установка громкости для указанного канала
- **Проверки**:
  - Громкость установлена корректно
  - Значение в пределах допустимого диапазона

### 4. UI-менеджер (scr_ui_manager.gml)

#### Тест 4.1: Инициализация UI-менеджера
- **Функция**: `scr_ui_manager_init()`
- **Входные данные**: none
- **Ожидаемый результат**: Подготовка системы UI
- **Проверки**:
  - Создание структуры данных UI
  - Установка начальных параметров отображения

#### Тест 4.2: Отображение UI элементов
- **Функция**: `scr_draw_ui_element(element_id, x, y)`
- **Входные данные**: ID элемента, координаты
- **Ожидаемый результат**: Правильное отображение UI элемента
- **Проверки**:
  - Элемент отображен в правильной позиции
  - Стиль элемента соответствует состоянию игры

#### Тест 4.3: Обработка ввода UI
- **Функция**: `scr_handle_ui_input(mouse_x, mouse_y, button)`
- **Входные данные**: координаты мыши, состояние кнопки
- **Ожидаемый результат**: Обработка взаимодействия с UI элементами
- **Проверки**:
  - Событие корректно обработано
  - Возвращается правильный результат взаимодействия

#### Тест 4.4: Обновление состояния UI
- **Функция**: `scr_update_ui_state(game_state)`
- **Входные данные**: текущее состояние игры
- **Ожидаемый результат**: Обновление отображения UI в соответствии с состоянием
- **Проверки**:
  - Правильные элементы отображаются
  - Элементы правильно реагируют на состояние игры

## Требования к тестированию

1. Каждый модуль должен быть протестирован изолированно
2. Все публичные функции должны быть протестированы
3. Должна быть обеспечена обработка граничных случаев
4. Тесты должны проверять как нормальное поведение, так и ошибочные ситуации

## Стратегия тестирования

1. Модульные тесты должны запускаться автоматически при сборке
2. Покрытие кода должно быть не менее 80%
3. Все критические функции должны быть протестированы
4. Тесты должны быть независимы друг от друга
