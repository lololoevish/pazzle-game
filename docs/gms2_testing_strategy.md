# Стратегия тестирования GMS2-версии

## Обзор

Этот документ описывает актуальную стратегию тестирования GameMaker Studio 2 версии `Adventure Puzzle Game`.

Текущий объект тестирования - рабочая ветка в `pazzle-game-gamemaker/` с маршрутом `Menu -> Town -> Cave 1..12 -> Victory -> Town`.

## Цели тестирования

1. Подтвердить, что 12-уровневая модель прогресса работает одинаково в runtime, сохранениях и UI.
2. Проверить, что хаб, NPC-активности, пазлы и победная сцена не ломают основной маршрут.
3. Ловить регрессии после изменений в `scr_game_state.gml`, `scr_save_system.gml`, `obj_player.gml`, `obj_npc.gml`, `scr_audio_manager.gml` и `scr_ui_manager.gml`.

## Слои проверки

### 1. Модульные сценарии

Проверяются отдельные GML-скрипты и сервисные функции:

- `scr_game_state.gml`
- `scr_save_system.gml`
- `scr_audio_manager.gml`
- `scr_ui_manager.gml`

Что важно проверить:

- корректную инициализацию новой игры;
- сохранение и восстановление 12 уровней;
- расчёт `completed`, `lever_pulled`, opened/completed progress и финала экспедиции;
- устойчивость UI и аудио helper-функций к безопасным вызовам.

### 2. Интеграционные сценарии

Проверяются связи между системами:

- `obj_game_manager` + `scr_game_state.gml`;
- `scr_save_system.gml` + глобальный `game_progress`;
- `obj_player` + переходы между уровнями;
- `obj_npc` + награды + сохранения;
- `scr_ui_manager.gml` + фактический прогресс экспедиции.

### 3. Функциональные сценарии

Проверяется пользовательский маршрут:

- новая игра;
- вход в хаб;
- запуск и завершение NPC-активностей;
- прохождение ранней, средней и поздней пещеры;
- возврат в хаб;
- сохранение и повторная загрузка;
- завершение 12-й пещеры и переход в `Victory`.

## Ручной regression checklist

Прогонять после каждого изменения, затрагивающего прогресс, комнаты, UI, аудио или NPC:

1. Запуск новой игры без сохранения.
2. Проверка, что в начале доступен только уровень 1.
3. Прохождение уровня 1 и открытие уровня 2.
4. Проверка любого уровня из диапазона 7-12 на корректную загрузку и возврат.
5. Разговор и успешное завершение активности у механика, архивариуса и старосты.
6. Сохранение игры и повторная загрузка.
7. Проверка, что прогресс HUD и цель экспедиции не сбрасываются.
8. Проверка, что после 12-го уровня экспедиция считается завершённой.
9. Проверка `reset_game()` / новой игры после существующего сохранения.
10. Просмотр output log на предмет ошибок и предупреждений, связанных с прогрессом, UI и пазлами.

## Smoke-набор перед коммитом крупных gameplay-изменений

- `test_game_state.gml`
- `test_save_system.gml`
- `test_audio_manager.gml`
- `test_ui_manager.gml`
- ручной проход: `menu -> town -> level 1 -> save/load -> town`

## Известные ограничения

- Автоматический линтинг `.gml` через `Biome` в текущем репозитории не работает, потому что эти файлы игнорируются конфигурацией.
- Часть старых справочных документов может содержать historical reference к ранней 6-уровневой модели; актуальный GMS2 scope закреплён в `docs/README.md`.
- Тестовые сценарии в GML являются полуавтоматическими: они полезны для проверки API и инвариантов, но не заменяют ручной runtime-проход в GameMaker.

## Критерии готовности после правок

Изменение считается безопасным, если одновременно выполняются условия:

- модульные тестовые сценарии под изменённые системы не падают;
- ручной smoke-проход не ломает маршрут новой игры;
- прогресс, сохранение и HUD согласованы между собой;
- output log не показывает новых ошибок по изменённым модулям.
