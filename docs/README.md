# Архитектура Adventure Puzzle Game

Этот документ является источником правды по текущей верхнеуровневой архитектуре проекта.

## Текущее состояние

- Основная рабочая версия проекта находится в `pazzle-game-gamemaker/`.
- Rust-ветка выступает как исторический референс маршрутов, механик и структуры прогресса.
- Текущий целевой продукт - одиночная 2D adventure/puzzle игра на GameMaker Studio 2 с хабом, 12 пещерами, побочными NPC-активностями, системой сохранений, аудио-слоем и VN-подачей диалогов.

## Структура проекта

### Основной игровой код

```text
pazzle-game-gamemaker/

│   ├── obj_game_manager.gml     # Общий цикл состояния игры

│   ├── obj_npc.gml              # NPC, мини-игры, диалоги, VN-интеграция

├── scripts/
│   ├── scr_game_state.gml       # Прогресс, уровень, цель, глобальные структуры
│   ├── scr_save_system.gml      # Сохранение и загрузка
│   ├── scr_audio_manager.gml    # Музыка, SFX, приоритеты, контексты
│   ├── scr_ui_manager.gml       # Общие UI-оверлеи и сервисные окна
```

Rust-ветка больше не является основной исполняемой целью проекта, но остаётся полезной как референс для маршрутов, UX-логики и старой реализации механик.

## Игровой маршрут

Текущий целевой маршрут GMS2-версии:

`Menu -> Town -> Cave 1..12 -> Victory -> Town`

Ключевые правила маршрута:

- хаб остаётся центральной точкой возврата и общения с NPC;
- прогресс по основной линии завязан на прохождение пещер и опускание рычагов;
- NPC-активности живут отдельно от основного маршрута и не должны ломать прогрессию пещер;
- повторный вход в уже решённую пещеру должен сохранять понятную и предсказуемую логику прогресса.

## Основные игровые системы

### 1. Система прогресса

- хранит состояние 12 уровней;
- различает `completed`, `lever_pulled` и связанные флаги побочного прогресса;
- определяет текущую цель экспедиции и завершённость маршрута;
- используется одновременно меню, хабом, пазлами и системой сохранений.

### 2. Система сохранений

- использует локальные JSON/INI-подобные сохранения внутри GMS2 runtime;
- сохраняет уровни, золото, предметы и статусы NPC-активностей;
- обязана оставаться совместимой с актуальной моделью прогресса на 12 уровней.

### 3. Хаб и NPC

- `Town` - это живая прослойка между пещерами, а не только экран выбора уровня;
- в хабе есть минимум 3 ключевых NPC с отдельными активностями и наградами;
- диалоги поддерживают реактивность по прогрессу и постепенный переход к VN-подаче;
- VN-система уже интегрирована на уровне скриптов и объекта NPC, но требует дальнейшей runtime-валидации и наполнения реальными портретами.

### 4. Набор головоломок

Поддерживаются 12 целевых пещерных активностей:

1. лабиринт;
2. поиск слов;
3. ритм/паттерн;
4. memory match;
5. платформер;
6. финальное испытание первой арки;
7. загадки;
8. звуковые ловушки;
9. прыгающий путь;
10. advanced memory;
11. песнь пещер;
12. эпический финал.

### 5. Аудио и сценическая подача

- музыка переключается по сценам и контекстам;
- есть базовые SFX для UI, пазлов и игровых событий;
- поддерживаются контекстные модификаторы громкости и приоритеты музыки;
- визуальный стиль движется в сторону более сценичной chapter-подачи для меню, хаба, пещер и финала.

## Ключевые архитектурные ограничения

- Проект всё ещё несёт переходный долг между старой 6-уровневой моделью и новой 12-уровневой структурой.
- Часть документации и часть runtime-логики до сих пор расходятся по тому, что считается завершённой миграцией.
- В GMS2-слое много процедурных скриптов большого размера, поэтому любое расширение должно по возможности сопровождаться декомпозицией ответственности.
- Автоматический линтинг и форматирование `.gml` сейчас отсутствуют; качество поддерживается через ревизию кода, сценарии тестирования и ручной smoke/regression-проход.

## Что считается актуальным scope проекта

Актуальный scope включает:

- рабочую GMS2-версию как главный продукт;
- 12-пещерную прогрессию;
- хаб с NPC-активностями, диалогами и наградами;
- сохранения и повторяемый маршрут прохождения;
- аудио и сценическую визуальную подачу;
- актуальную документацию и Memory Bank, синхронизированные с кодом.

## Что ещё не считается полностью закрытым

- полная runtime-стабильность GMS2-версии;
- унификация всей логики прогресса на 12 уровней без legacy-веток;
- актуальный regression checklist и доказуемый smoke-проход по основному маршруту;
- release-ready состояние документации, ассетов и финальной полировки.
